# Review: Task 024

## Round 1

- [ ] **Resume accuracy regression in `register_sources`**: On resume, previously-completed jobs are not reflected in the progress dashboard — all bars start at 0. The old `set_data_source("k-extract", total_jobs, total_pending)` computed `completed_jobs = total - pending`, so on resume the bars correctly showed prior progress (e.g., `200/714` if 200 jobs were already completed). The new `register_sources(per_source_totals)` (`progress.py:105`) only accepts total counts and initializes all `SourceProgress` instances with `completed=0, failed=0` (`progress.py:113-114`). In the orchestrator (`orchestrator.py:488`), only `per_source_totals` (total job counts) are passed — no per-source completed or pending counts. As a result, on a resumed run where 200 of 714 jobs were already completed: the bars start at `0/714` and end at `514/714`, never reaching 100%. Both the spec (`extraction-pipeline.md:199`: "the dashboard can show accurate totals from the start") and the task description (same wording) require accurate initial counts. The `register_sources` API needs to accept initial completed/failed counts per source, and the orchestrator needs to query and pass per-source completed counts on resume.
