# Review: Task 007

## Round 1

- [process-revision-complete] **search_relationships missing "List All" mode (spec section 2, task description item 2).** The task description lists "List All: all instances of a type" as a required feature. The spec defines three modes in a table: Type Definition, List by Slug, and List All. The "List All" mode takes "A relationship type (optionally filtered by slug)" and returns "All instances of the given type." The implementation at `src/k_extract/extraction/tools.py:273` enters Type Definition mode whenever `slug is None`, making it impossible to retrieve all instances of a relationship type without providing a slug. There is no input parameter or code path to distinguish "give me the type definition" from "give me all instances." The test `TestSearchRelationshipsListAll::test_list_all_via_show_all` at `tests/extraction/test_tools.py:629` only tests `show_all=True` WITH a slug (which is just uncapped List by Slug, not true List All).

- [process-revision-complete] **search_relationships slug-based lookups only search the first matching composite key (spec section 2, "List by Slug" mode).** At `src/k_extract/extraction/tools.py:300`, when a forward type like "REFERENCES" matches multiple composite keys (e.g., `Product|REFERENCES|Product` and `Component|REFERENCES|Service`), the implementation arbitrarily uses `composite_keys[0]` for all slug-based lookups (single-slug at line 317 and two-slug at line 302). The spec says "List by Slug" returns "Relationship instances involving the slug(s)" — it does not restrict this to a single composite key. An agent querying `{"relationship_type": "REFERENCES", "slug": "component:foo"}` would only get results from the first matching composite key, silently missing relationships under other matching keys.

- [process-revision-complete] **search_entities "Get by Slugs" mode returns kebab-case entity_type instead of PascalCase (spec section 1, "Get by Slugs" mode).** The spec says results "include entity_type when type is auto-resolved." Throughout the system API, entity types are PascalCase: `EntityTypeDefinition.type` is validated PascalCase, `ManageEntityInput.entity_type` expects PascalCase, `SearchEntitiesInput.entity_type` expects PascalCase. However, `_entity_to_dict` at `src/k_extract/extraction/tools.py:40-43` returns `entity.entity_type` which is the kebab-case slug prefix (e.g., "product" not "Product"). The test at `tests/extraction/test_tools.py:244` confirms this: `assert data[0]["entity_type"] == "product"`. An agent receiving "product" from search_entities cannot directly pass it to manage_entity or search_entities type definition mode, which both require "Product" (PascalCase).

## Round 2

- [process-revision-complete] **manage_entity response returns kebab-case entity_type — incomplete fix from Round 1 finding 3 (spec section 3, process checklist item 24).** The Round 1 fix added an `ontology` parameter to `_entity_to_dict` and updated the two `search_entities` call sites (lines 167 and 177) to pass `ontology`, resolving entity_type to PascalCase. However, `manage_entity` at `src/k_extract/extraction/tools.py:442` still calls `_entity_to_dict(updated)` without the `ontology` parameter. The `ontology` object is available in the closure (captured from the `create_extraction_tools` factory at line 133) but is not passed. As a result, `search_entities` returns `"entity_type": "Product"` (PascalCase) while `manage_entity` returns `"entity_type": "product"` (kebab-case). An agent that reads the `entity_type` from a `manage_entity` response and passes it to `search_entities` type definition mode (which calls `ontology.get_entity_type(entity_type)` keyed by PascalCase) would get an "Unknown entity type" error. This is exactly the cross-tool format inconsistency that process checklist item 24 was designed to catch.
