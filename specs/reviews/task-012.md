# Review: Task 012

## Round 1

- [ ] **Output file not cleared on `--force` fresh start.** `JsonlWriter` always opens files in append mode (`"a"`, `writer.py:41,54`). On a `--force` fresh start, the orchestrator deletes all jobs from the database (`orchestrator.py:213`) and stores a new fingerprint, but never truncates or deletes the existing JSONL output file (`orchestrator.py:230`). Old CREATEs from the discarded run persist in the file, followed by new DEFINEs and new CREATEs. This produces an inconsistent output file with stale data mixed into the fresh run's output. Spec reference: `extraction-pipeline.md` — `--force` "Discard previous state and start fresh"; `output-format.md` — "DEFINE… Must appear before any CREATE/UPDATE/DELETE."

- [ ] **`start_order` calculation incorrect when max order is 0.** At `orchestrator.py:274`, `start_order = (max_order or -1) + 1`. The `COALESCE(MAX("order"), -1)` query returns `0` when the first data source's jobs start at order 0. Python's `or` operator treats `0` as falsy, so `0 or -1` evaluates to `-1`, producing `start_order = 0`. This causes the second data source's jobs to start at order 0, colliding with the first source's order=0 job. The `or -1` fallback is redundant (COALESCE already handles the null case) and introduces the bug. Fix: `start_order = max_order + 1`. Spec reference: `job-lifecycle.md` — "`order`: Global ordering number across all data sources (determines processing sequence)."

- [ ] **CLI prints "Re-run to retry failed jobs" but re-running does not retry failed jobs.** `run.py:106` outputs `"Re-run to retry failed jobs."` `claim_next_job` (`jobs.py:134-198`) only claims jobs with status `PENDING`. Failed jobs remain in `FAILED` state and are never reset to `PENDING` on re-run. The user is told to re-run but doing so has no effect on failed jobs. Either the orchestrator should reset `FAILED` jobs to `PENDING` at startup, or the message should direct the user to a manual reset command. Spec reference: `extraction-pipeline.md:209` — `"Re-run to retry failed jobs, or use k-extract jobs --config extraction.yaml to inspect."` shows re-run as the retry mechanism.

- [ ] **`max_jobs` cap can be exceeded by up to (workers−1) jobs.** The shared counter is checked at the top of the worker loop (`worker.py:87-90`) BEFORE claiming a job, but incremented AFTER `await run_agent()` completes (`worker.py:136-137`). With N concurrent workers launched via `asyncio.gather` (`orchestrator.py:372`), all N workers can pass the check (counter still 0), claim a job (synchronous), and enter `await run_agent()` before any worker increments the counter. Example: `max_jobs=1` with 3 workers → all 3 pass the check, claim a job, and process it. Result: 3 jobs processed when cap is 1. The counter should be incremented immediately after claiming the job (before `await run_agent`). Spec reference: `extraction-pipeline.md` — "A shared counter tracks total jobs processed across all workers for enforcing global job limits."

- [ ] **Resume uses timeout-based stale job detection; recently-crashed jobs are not recovered.** On resume, the orchestrator calls `reset_stale_jobs(session)` with a 60-minute timeout (`orchestrator.py:221-223`). If a previous run crashed less than 60 minutes ago, in-progress jobs from that run won't be reset to pending — they remain stuck in `in_progress` state. No worker can claim them (they're not `PENDING`), and no timeout-based scanner runs during the extraction. The codebase provides `reset_all_in_progress` (`jobs.py:259-277`) which unconditionally resets all in-progress jobs — appropriate at run startup since no workers are actively running. Spec reference: `job-lifecycle.md` — "Unconditional reset at startup: All in_progress jobs are reset to pending at the start of each extraction run. This is appropriate when each run is expected to start fresh; any job still marked in_progress from a previous run is considered stale."
