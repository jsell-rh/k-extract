# Review: Task 012

## Round 1

- [process-revision-complete] **Output file not cleared on `--force` fresh start.** `JsonlWriter` always opens files in append mode (`"a"`, `writer.py:41,54`). On a `--force` fresh start, the orchestrator deletes all jobs from the database (`orchestrator.py:213`) and stores a new fingerprint, but never truncates or deletes the existing JSONL output file (`orchestrator.py:230`). Old CREATEs from the discarded run persist in the file, followed by new DEFINEs and new CREATEs. This produces an inconsistent output file with stale data mixed into the fresh run's output. Spec reference: `extraction-pipeline.md` — `--force` "Discard previous state and start fresh"; `output-format.md` — "DEFINE… Must appear before any CREATE/UPDATE/DELETE."

- [process-revision-complete] **`start_order` calculation incorrect when max order is 0.** At `orchestrator.py:274`, `start_order = (max_order or -1) + 1`. The `COALESCE(MAX("order"), -1)` query returns `0` when the first data source's jobs start at order 0. Python's `or` operator treats `0` as falsy, so `0 or -1` evaluates to `-1`, producing `start_order = 0`. This causes the second data source's jobs to start at order 0, colliding with the first source's order=0 job. The `or -1` fallback is redundant (COALESCE already handles the null case) and introduces the bug. Fix: `start_order = max_order + 1`. Spec reference: `job-lifecycle.md` — "`order`: Global ordering number across all data sources (determines processing sequence)."

- [process-revision-complete] **CLI prints "Re-run to retry failed jobs" but re-running does not retry failed jobs.** `run.py:106` outputs `"Re-run to retry failed jobs."` `claim_next_job` (`jobs.py:134-198`) only claims jobs with status `PENDING`. Failed jobs remain in `FAILED` state and are never reset to `PENDING` on re-run. The user is told to re-run but doing so has no effect on failed jobs. Either the orchestrator should reset `FAILED` jobs to `PENDING` at startup, or the message should direct the user to a manual reset command. Spec reference: `extraction-pipeline.md:209` — `"Re-run to retry failed jobs, or use k-extract jobs --config extraction.yaml to inspect."` shows re-run as the retry mechanism.

- [process-revision-complete] **`max_jobs` cap can be exceeded by up to (workers−1) jobs.** The shared counter is checked at the top of the worker loop (`worker.py:87-90`) BEFORE claiming a job, but incremented AFTER `await run_agent()` completes (`worker.py:136-137`). With N concurrent workers launched via `asyncio.gather` (`orchestrator.py:372`), all N workers can pass the check (counter still 0), claim a job (synchronous), and enter `await run_agent()` before any worker increments the counter. Example: `max_jobs=1` with 3 workers → all 3 pass the check, claim a job, and process it. Result: 3 jobs processed when cap is 1. The counter should be incremented immediately after claiming the job (before `await run_agent`). Spec reference: `extraction-pipeline.md` — "A shared counter tracks total jobs processed across all workers for enforcing global job limits."

- [process-revision-complete] **Resume uses timeout-based stale job detection; recently-crashed jobs are not recovered.** On resume, the orchestrator calls `reset_stale_jobs(session)` with a 60-minute timeout (`orchestrator.py:221-223`). If a previous run crashed less than 60 minutes ago, in-progress jobs from that run won't be reset to pending — they remain stuck in `in_progress` state. No worker can claim them (they're not `PENDING`), and no timeout-based scanner runs during the extraction. The codebase provides `reset_all_in_progress` (`jobs.py:259-277`) which unconditionally resets all in-progress jobs — appropriate at run startup since no workers are actively running. Spec reference: `job-lifecycle.md` — "Unconditional reset at startup: All in_progress jobs are reset to pending at the start of each extraction run. This is appropriate when each run is expected to start fresh; any job still marked in_progress from a previous run is considered stale."

## Round 2

- [ ] **Resume summary understates completed job count for partially-completed data sources.** At `orchestrator.py:318-344`, previously completed jobs are only counted when `source_pending == 0` (the `already_completed` query runs inside that branch). When a data source has both completed and pending jobs — the normal resume scenario after some jobs succeeded and some failed — the previously completed jobs are never added to `total_completed`. Example: source has 10 jobs, 7 completed in run 1, 3 failed. On resume, `reset_failed_jobs` sets the 3 failed to pending. `source_pending = 3`, so the `source_pending == 0` branch is skipped. Workers process 3 jobs and `total_completed += 3` from worker results. The 7 already-completed jobs are not counted. Summary reports "3/10 jobs completed" instead of "10/10 jobs completed." Spec reference: `extraction-pipeline.md:199-200` — "Extraction complete. 107/110 jobs completed, 3 failed." Acceptance criteria: "Completion summary with stats, cost, and failed job details."

- [ ] **Conversation log filenames are per-worker, not per-worker-per-job.** `ConversationLogger.__init__` at `logging.py:71` creates filename `worker-{worker_id}.jsonl` — one file per worker. All conversation data from every job processed by that worker is appended to the same file. The `job_id` is available in `run_agent()` (`agent.py:133`) but is not passed to `ConversationLogger`. The spec shows per-job files: `worker_01_batch_0001.jsonl`, `worker_01_batch_0002.jsonl`, etc. (`extraction-pipeline.md:233-236`). With per-worker files, there is no way to isolate the conversation for a specific job — the stated purposes of "post-hoc debugging of extraction quality issues" and "reproducing agent behavior for prompt tuning" (`extraction-pipeline.md:241-243`) require per-job granularity.

- [ ] **`asyncio.gather` without `return_exceptions=True` violates failure isolation.** At `orchestrator.py:381`, `await asyncio.gather(*worker_tasks)` is called without `return_exceptions=True`. If any worker raises an unhandled exception, `asyncio.gather` propagates it immediately and cancels all remaining worker tasks. The `worker_loop` function (`worker.py:85-173`) has no top-level exception handler; exceptions from `claim_next_job` (DB error), `mark_completed`/`mark_failed` (DB error), `store.clear_staging` (DB error), `store.pop_committed`, or `writer.write_operations` (I/O error) propagate unhandled. A single infrastructure error in one worker would terminate all concurrent workers. Spec reference: `extraction-pipeline.md:167-168` — "The failure is isolated: other workers (whether concurrent tasks or parallel processes) are unaffected and continue processing their own jobs." Acceptance criteria: "Job failure isolation (one failure doesn't affect other workers)."
