# Review: Task 020

## Round 1

- [process-revision-complete] **PipelineProgress is created per data source instead of at pipeline start.** The task description (section 2) specifies: "Create a `PipelineProgress` instance at pipeline start." The implementation creates a new `PipelineProgress(worker_count)` inside the `for ds in config.data_sources:` loop (`orchestrator.py:413`), meaning a fresh instance is constructed for each data source. This causes two fields that should be cumulative across the entire pipeline run to reset per data source: (1) `elapsed_seconds` — the task says "Elapsed wall-clock time (from pipeline start)", but `_start_time` resets when a new `PipelineProgress` is instantiated for each data source; (2) `cumulative_cost` — the task says "Cumulative cost (updated as workers finish jobs)", but cost resets to `0.0` per data source. The `set_data_source()` method was designed for exactly this scenario — it resets per-source fields (total, pending, completed, failed, data source name) while preserving cumulative fields (cost, elapsed time) — but it is never used on a persistent instance. A single `PipelineProgress` should be created before the data source loop and reused across sources.
- [process-revision-complete] **Setup spinner does not cover job generation as specified.** Task section 4 says: "Show a spinner during the setup phase (config loading, fingerprinting, resume evaluation, job generation) before workers start." The spinner starts at `orchestrator.py:170` and stops at `orchestrator.py:284-285`, covering steps 1-11 (config loading, DB setup, fingerprinting, resume evaluation). However, job generation occurs at `orchestrator.py:317-339`, inside the data source loop, after the spinner has already stopped. Users see no loading indicator during job generation, which can involve file discovery and batching computation for large data sources.

## Round 2

- [process-revision-complete] **All three `Live` displays lack exception-safe cleanup (no try/finally).** The task (section 2) specifies: "Wrap the worker execution phase in a `rich.live.Live` context." The word "context" implies proper resource lifecycle management (context manager or equivalent try/finally). All three `Live` instances in `orchestrator.py` use manual `.start()` / `.stop()` without try/finally: (1) `setup_live` at lines 165–285 — if `load_config()`, `discover_model_capabilities()`, `compute_fingerprint()`, or any other step between `.start()` and `.stop()` raises, `setup_live.stop()` is never called (the `HARD_STOP` case at line 233 is handled, but other exceptions are not); (2) `job_gen_live` at lines 318–358 — if `discover_files()`, `create_jobs()`, or DB operations raise, the spinner is never stopped; (3) the dashboard `live` at lines 438–476 — if the main task is cancelled (e.g., `KeyboardInterrupt` → `asyncio.CancelledError` propagates through `asyncio.gather`), the code after gather (including `live.stop()` at line 476) never executes. Since `Live.stop()` restores the terminal cursor and exits Rich's screen management, skipping it leaves the terminal with a hidden cursor or in an inconsistent state. This is a real user-experience issue for a CLI tool where the task description says users may stare at the terminal "for minutes to hours" — Ctrl+C to abort is a common scenario. Rich's `Live` is designed to be used as a context manager (`with Live(...) as live:`) for exactly this reason; the manual `.start()` / `.stop()` pattern requires try/finally for equivalence.
