# Review: Task 002

## Round 1

- [process-revision-complete] `_pascal_to_kebab` fails for entity types with consecutive uppercase letters (acronyms). `ontology.py:295-302` uses `re.sub(r"([a-z0-9])([A-Z])", ...)` which only inserts hyphens between a lowercase/digit and an uppercase letter. For "SREFile" (referenced in spec section 1.5 as an example entity type), `_pascal_to_kebab("SREFile")` returns `"srefile"` instead of `"sre-file"`. This breaks `find_entity_type_for_slug` and `get_entities_by_type` for any entity type containing acronyms. No test covers this case.
- [process-revision-complete] Missing `entities` dict key-to-slug consistency validator on the `Ontology` model. `ontology.py:125-146` has model validators ensuring `entity_types` dict keys match their `type` field and `relationship_types` dict keys match their `composite_key`, but there is no corresponding validator for the `entities` dict (line 122). Constructing `Ontology(entities={"wrong-key": EntityInstance(slug="product:test", properties={})})` silently succeeds, after which `get_entity_by_slug("product:test")` returns `None` despite the entity existing in the dict. Spec section 2.2 requires slugs to be the primary identifier; the inconsistency between key and slug breaks all slug-based lookups.
- [process-revision-complete] No enforcement of relationship instance uniqueness. Spec section 3.2 states: "The `(source_slug, target_slug)` pair is unique within a composite key." `Ontology.relationships` is a plain `list[RelationshipInstance]` (line 123) with no uniqueness constraint. Neither the container nor `validate_relationship` (lines 237-276) checks for duplicate `(composite_key, source_slug, target_slug)` triples, so the same relationship can be inserted multiple times without error.
- [process-revision-complete] `validate_entity` does not enforce structural type protection. Spec section 4.4 states: "Validation rejects edits to protected types." The `Ontology` model provides query helpers (`EntityTypeDefinition.is_structural` at line 65, `Ontology.is_structural_entity_type` at line 278), but `validate_entity` (lines 186-235) never consults them. An entity belonging to a structural-tier type (e.g., `DataSource` with `tier=structural`) passes `validate_entity` without any protection error. The task acceptance criteria explicitly lists "Structural type protection (tier-based editability check)."
