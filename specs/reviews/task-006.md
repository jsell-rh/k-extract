# Review: Task 006

## Round 1

- [process-revision-complete] **Relationship properties not merged during validate-and-commit** (`store.py:460-469`). When building the merged relationship view, staged relationships fully replace shared relationships: `rel_map[key] = rel`. Compare with entity merge (lines 447-458) which correctly does `{**merged_entities[slug].properties, **props}`. If a shared relationship has properties `{context: "old", weight: 5}` and a staged relationship has `{context: "new"}`, the merged result is `{context: "new"}` — the `weight` property is lost. Spec reference: domain-model.md Section 5.4 ("merge property changes"), agent-tools.md Section 4 Edit Mode step 4 ("Deep-copy existing, merge changes into properties").

- [process-revision-complete] **Relationship properties not merged in virtual view read path** (`store.py:629-663`). `_merged_relationships_by_key` replaces shared relationships with staged ones: `rels[key] = _row_to_relationship(row)` (line 663). Compare with `_merged_entities_by_type` (lines 580-591) which correctly merges entity properties. The virtual view should show a property-merged view per domain-model.md Section 5.2 ("merged view of shared ontology + pending edits") and Section 5.4. This same replacement-instead-of-merge pattern also affects `search_relationships_by_type` and `search_relationships_by_slug` which both delegate to this helper.

## Round 2

- [process-revision-complete] **Missing structural protection for staged relationship types in validate-and-commit** (`store.py:696-812`). The `_validate_merged` method checks structural protection for staged entity types (lines 711-718) using `type_def.is_structural`, but performs no structural protection check for staged relationship types. Three specs consistently require this: domain-model.md Section 4.4 ("Structural entity types **and relationship types** are protected from agent modification. Validation rejects edits to protected types."), agent-tools.md Section 5 step 2 ("Structural type guard: Staged edits must not include structural entity types **or structural relationship types**."), and concurrency-model.md Section 5 step 5 ("Structural relationships not modified"). The root cause is that `RelationshipTypeDefinition` (`ontology.py:70-99`) lacks any structural/tier classification — unlike `EntityTypeDefinition` which has `tier: Tier` and an `is_structural` property. Without this classification, the store cannot distinguish structural relationship types (e.g., CONTAINS, HAS_ROOT_FOLDER per agent-tools.md Section 4) from agent-managed types (e.g., REFERENCES). A worker can currently stage and commit structural relationships without validation catching the violation. The domain model's `RelationshipTypeDefinition` needs a category/tier field (analogous to domain-model.md Section 1.6's three categories: structural, agent-managed, aggregator-managed), and `_validate_merged` needs to check staged relationship keys against it.
