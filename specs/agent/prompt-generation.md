# Prompt Composition

Prompts are composed during `k-extract init` and stored in the config file. At run time, no LLM calls are needed for prompt composition — everything is pre-baked in the config with variable placeholders for per-job substitution.

## Composition Layers

### Layer 1: Static Template (shipped with k-extract)

The "how to be an extraction agent" boilerplate. This is the same regardless of ontology or data source:

- Role definition (you are a knowledge extraction agent)
- Workflow steps (read files, extract entities/relationships, validate and commit)
- Tool usage rules (how to use search, manage, commit tools)
- Efficiency constraints (don't narrate, work autonomously)
- Quality constraints (don't create duplicates, verify before creating, use consistent slugs)
- **Deduplication instructions:** "Before creating any entity or relationship, search for it by slug. If it already exists, use the existing one — do not create a duplicate. The tools enforce this at the tool level, but searching first avoids unnecessary tool calls and helps you decide whether to create or update."

Lives as a template file in the k-extract package — not hardcoded in Python. This is an implementation detail of `init`; by the time the config exists, the template content has been composed into the final prompts.

### Layer 2: LLM-Generated Guidance (produced during `init`)

The "what to extract" instructions, generated by an LLM from the confirmed ontology + problem statement:

- Entity type extraction guidance: what each type represents, when to create one, what properties to capture
- Relationship type extraction guidance: what connections to look for, directionality, properties
- Problem-statement-driven priorities: what to focus on, what level of granularity, what to ignore

This is where the problem statement shapes extraction behavior. Translating an ontology schema + problem statement into natural language extraction guidance is itself a language task — hence LLM generation.

### Layer 3: Per-Job Variables (substituted at run time)

The "what files to process" details, substituted via template variables at run time:

- `{job_id}` — job identifier
- `{file_count}` — number of files in the job
- `{total_characters}` — total character count
- `{file_list}` — list of file paths to process
- Other job-specific metadata

Pure string substitution, no LLM call.

## Config File Structure

The composed prompts are stored in the config as separate fields:

```yaml
prompts:
  system_prompt: |
    # Role
    You are a knowledge extraction agent...
    
    # Workflow
    1. Read your assigned files
    2. Extract entities and relationships using the provided tools
    3. Validate and commit your changes
    
    # Extraction Guidance
    ## Entity Types
    - TestCase: Represents an individual test. Create one for each test function...
    - Component: Represents a software component or module...
    
    ## Relationship Types
    - TESTS: Connect a TestCase to the Component it exercises...
    
    # Tool Usage
    ...
    
    # Quality Rules
    ...

  job_description_template: |
    ## Job {job_id}
    
    Process the following {file_count} files ({total_characters} characters):
    
    {file_list}
    
    Extract all entities and relationships from these files according to your instructions.
```

### Why Separate Fields

- The user can edit the extraction guidance without touching the workflow boilerplate
- The job description template can be tweaked independently (e.g., adding instructions about a specific batch)
- Both are inspectable: the user can read the config and see exactly what instructions agents will receive

### Why in the Config (Not the DB)

- **Reproducibility**: the config file is a complete, portable artifact. Share it and someone else can reproduce the extraction.
- **Inspectability**: YAML is human-readable. Users can review and edit prompts before running.
- **Fingerprinting**: the config is part of the environment fingerprint hash (see [extraction-pipeline.md](../process/extraction-pipeline.md#resumability-and-environment-fingerprinting)). Editing prompts changes the fingerprint, which correctly prevents resuming with mixed old/new results.
- **Version control**: the config can be committed to git, enabling diff-based review of prompt changes over time.

## Composition Flow

```
k-extract init /path/to/data
  1. Guided session → problem statement + ontology confirmed
  2. Load static template from k-extract package
  3. LLM generates extraction guidance from ontology + problem statement
  4. Compose: template + generated guidance → system_prompt
  5. Compose: job template with variable placeholders → job_description_template
  6. Write config file with both prompts as separate fields

k-extract run --config extraction.yaml
  1. Read system_prompt from config (used as-is)
  2. For each job: substitute variables into job_description_template
  3. Pass both to the agent SDK query()
```

## Relationship to Original Codebase

In the original system, system-level instructions, ontology-specific instructions, and job-specific instructions were all hand-written static files. In k-extract, they are generated from the config file. The system-level instructions may still have a static template component, but the ontology and job instructions are fully dynamic.
